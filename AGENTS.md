# AGENTS.md, 스킬 라이브러리 운영 기준

이 문서는 이 폴더에서 카드를 다루는 모든 AI 에이전트(Claude Code, Codex, Gemini 등)가 따라야 하는 단일 운영 기준이다. 도구가 무엇이든 규칙은 여기 하나만 본다.

## 문서 체계

| 문서 | 역할 | 갱신 시점 |
|---|---|---|
| `AGENTS.md` | 운영 기준의 정본. 카드 선정·제거·동기화·검증 규칙 전체 | 규칙이 바뀔 때만 |
| `MEMORY.md` | 결정 로그. 규칙이 왜 이렇게 되었는지, 미결 사항 | 운영 결정이 내려질 때마다 |
| `CLAUDE.md` | 이 문서로 안내하는 리다이렉트 | 원칙적으로 고정 |
| `.claude/commands/*.md` | 실행 절차(add-skill, find-skill, discovery-skill) | 절차가 바뀔 때 |

규칙(무엇을 받고 무엇을 거르나)은 이 문서가 정본이고, 절차(어떤 명령을 어떤 순서로 치나)는 커맨드 파일이 정본이다. 둘이 어긋나면 규칙은 이 문서를 따르고, 커맨드 파일을 이 문서에 맞게 고친다.

## 1. 프로젝트 정체성

**비개발자 비즈니스 리더를 위해 엄선한 Claude 스킬·플러그인 카탈로그**다. 모든 판단의 최종 질문은 하나다. "코드를 안 쓰는 경영 리더가 이 카드를 보고 당일 업무에 쓸 수 있는가?"

- 타깃: HR·영업·법무·마케팅·재무·전략·기획 등 비개발 직군의 리더와 지식 노동자.
- 반(反)타깃: 개발 워크플로우(PR, CI/CD, 테스트, 배포, 코드 리뷰)가 중심인 도구. 별점이 아무리 높아도 받지 않는다.
- 데이터 정본: `docs/skills.json` 하나. 3개 메타 카테고리, 13개 세부 카테고리.

## 2. 총량 룰: 150개 고정

카드 총량은 **150개로 고정**한다.

- 새 카드를 추가하면 반드시 기존 카드 하나를 제거해 150으로 되돌린다. 추가만 하고 끝내지 않는다.
- 총량 자체를 늘리거나 줄이는 변경은 사용자가 명시적으로 요청할 때만 한다.
- 왜 고정하는가: 총량 제한이 없으면 큐레이션이 수집으로 변질된다. 150이라는 상한이 "이 카드가 기존 카드보다 나은가"라는 비교 질문을 강제하고, 그 비교가 라이브러리의 품질을 지킨다.

## 3. 카드 입고 기준 (5개 게이트)

새 카드는 아래 게이트를 순서대로 통과해야 한다. 하나라도 걸리면 중단하고 사용자에게 알린다. 실행 절차는 `.claude/commands/add-skill.md`를 따른다.

### 게이트 1. 중복 검사

- `skills.json`의 `repo` 필드와 대소문자 무시로 정확 일치 대조. 일치하면 즉시 중단.
- repo가 달라도 `name`(Title Case 표시명)이 기존 카드와 같으면 중단하고 어떤 카드와 겹치는지 알린다.

### 게이트 2. 카테고리 검증

- 카테고리 id 목록은 런타임에 `skills.json`의 `categories[].id`에서 읽는다. 문서나 코드에 하드코딩하지 않는다.
- README 본문, repo description, topics를 근거로 카테고리 정의(`categories[].desc`)와 매칭한다.
- 사용자가 준 카테고리와 판단이 다르면 근거 2~3줄과 함께 역제안하고, 사용자가 고른 뒤에만 진행한다.

### 게이트 3. 비즈니스 리더 적합성

**적합 신호** (한 개 이상이면 적합 쪽):

- 비개발 도메인(HR·영업·법무·마케팅·재무·전략·기획)에 직접 쓸 수 있다.
- 문서·회의록·리서치·커뮤니케이션·의사결정 보조 같은 범용 지식 작업을 돕는다.
- 설치·설정이 README 기준 몇 단계 이내이고, YAML·코드 작성 없이 쓸 수 있다.
- 이미 만들어진 프리셋·템플릿을 골라서 쓰는 구조다.

**부적합 신호** (한 개라도 강하게 걸리면 부적합 쪽):

- README 핵심 키워드가 PR 생성·CI/CD·테스트·배포·모니터링·코드 리뷰·git hook·병합 등 개발 워크플로우에 한정된다.
- 코딩 에이전트 관리(칸반, 작업 할당, 서브에이전트 오케스트레이션)가 주 용도다.
- 사용에 YAML/JSON 워크플로우 작성, 서버 직접 구성, 조건분기·재시도 같은 엔지니어 멘탈모델이 필요하다.
- "풀스택", "파이프라인", "런타임", "멀티 LLM 오케스트레이션"이 중심 가치다.
- 도메인 전문성을 전제하는데 그 타깃이 리더가 아닌 실무자(개발자·분석가)다.

**판단 시 주의**:

- 별점이 높다고 자동 통과시키지 않는다. 스타는 많아도 개발자 전용인 저장소가 실제로 있다.
- README에 "누구를 위한 것인가" 섹션이 있으면 그 문구를 1순위 근거로 인용한다.
- 근거는 구체적으로 쓴다. "개발자용 같음"이 아니라 어떤 기능·어떤 용어에서 비개발자가 막히는지 적는다.
- 부적합·애매 판정이면 중단하고 근거와 함께 사용자에게 추가·보류·재분류를 묻는다.

### 게이트 4. MCP 서버 제외 (2026-07-17 결정)

**MCP 서버는 카드로 받지 않는다.** 저장소의 이름·설명·README가 MCP 서버 구축이나 제공을 핵심 정체성으로 삼으면 탈락이다.

- 왜: 2026-07-17에 MCP 서버 카드 25개를 전량 삭제했다. 이 라이브러리는 "골라서 바로 쓰는 스킬·플러그인" 카탈로그이고, 서버 설치·구성이 필요한 MCP는 비개발자 리더의 사용 장면과 맞지 않는다는 판단이다. 경위는 `MEMORY.md` 참조.
- 허용 범위: 스킬·플러그인이 본체이고 MCP 연동을 부가 기능으로 언급하는 정도는 받는다. 판단 기준은 "MCP를 빼도 이 카드의 가치가 성립하는가"다.

### 게이트 5. 에디터픽 평가 (선택 배지, 깐깐하게)

`editors_pick`은 흔해지면 가치를 잃는다. 기본값은 생략(false)이고, 아래 3축 평가를 통과한 후보만 사용자 승인을 받아 `true`로 단다.

**3축, 각 5점 만점**:

| 축 | 5점 | 4점 | 3점 | 2점 | 0~1점 |
|---|---|---|---|---|---|
| **효용성** (리더가 얼마나 빨리 효용을 뽑나) | 설치 즉시 당일 투입, 반복 사용 명확 | 설정 1~2단계 후 곧바로 가치 | 세팅 여러 단계, 허들 있음 | 특정 시나리오만 | 개발자 멘탈모델 요구 |
| **대표성** (카테고리 내 위상) | 카테고리 대표, 상위 대안 없음 | 명확한 차별점·우위 | 차별점 있으나 경쟁과 비등 | 비슷한 카드 다수 | 동급·상위 카드 다수 |
| **신뢰성** (저장소 신뢰도) | 공식 조직급 + ★수천 + 활발한 유지보수 | ★1k+ AND 검증된 저자 | ★1k+ 또는 검증된 저자 | ★500~1k, 저자 미검증 | ★<500 또는 정체 |

- 최근 업데이트가 1년 이상 정체면 신뢰성 -1. README에 deprecated 명시면 신뢰성 1점 이하.
- **평균 4.0 이상이면 PICK 후보**. 과락 규칙은 없다.
- 5점은 "이 카테고리에서 최상위 대안이 없다" 수준일 때만. 인플레 금지.
- 망설여지면 탈락이다. "의심스러우면 PICK 아님"이 원칙.
- 같은 카테고리에 PICK이 이미 2개 이상이면 신규는 평균 4.3+ 권장으로 더 엄격하게.
- PICK 후보면 3축 점수와 근거를 보여주고 사용자 승인(y/n)을 받은 뒤에만 `true`로 넣는다.

## 4. 카드 제거 기준

150 복원을 위해 하나를 제거할 때 이 순서로 고른다.

1. **자체 카드는 보호한다.** `airoasting/*` 저장소는 별점이 낮아도 제거 후보에서 뺀다. 이 라이브러리 운영자의 자체 자산이기 때문이다.
2. 남은 서드파티 카드 중 **별점(stars) 최하위부터** 제거한다. 별점이 같으면 fork 수가 낮은 쪽을 뺀다.
3. 어떤 카드를 지울지 애매하면 상위 후보 2개를 제시하고 사용자에게 확인받는다.

일괄 삭제(카테고리 단위, 유형 단위)는 총량을 바꾸므로 사용자의 명시 요청이 있을 때만 하고, 삭제로 고아가 되는 `authors` 엔트리도 함께 정리한다.

## 5. 외부 발굴 기준 (discovery)

라이브러리에 없는 후보를 찾을 때는 `.claude/commands/discovery-skill.md` 절차를 따르되, 판단 기준은 다음과 같다.

- 게이트 3(리더 적합성)과 게이트 4(MCP 제외)를 사전 필터로 먼저 적용한다. 통과 못 한 후보는 점수 평가 자체를 생략한다.
- 살아남은 후보에 게이트 5와 같은 3축 평가를 적용하고, **README를 실제로 읽은 근거**와 함께 점수를 노출한다. 추측으로 점수를 만들지 않는다.
- 평균 3.5 미만은 추천하지 않는다. 추천 강도: 4.5+ 강력 추천, 4.0~4.4 추천, 3.5~3.9 검토.
- 발굴은 추천까지만이다. 카드 추가는 반드시 add-skill 게이트를 다시 통과한다.

## 6. 카드 스키마와 필드 규칙

`skills.json`의 `skills` 배열 마지막에 추가하며 필드 순서를 고정한다.

```json
{
  "category": "<검증된 카테고리 id>",
  "name": "<repo name을 Title Case로>",
  "repo": "<owner>/<repo>",
  "author": "<owner>",
  "stars": 0,
  "added_at": "YYYY-MM-DD",
  "desc": "<한국어 2문장>",
  "tags": ["#tag1", "#tag2"]
}
```

- `desc`: 한국어 **정확히 2문장**, 합쳐서 **100~160자**(공백 포함). 담백한 설명체, "~합니다" 종결. 과장·이모지·느낌표 금지.
  - 1문장: "이 스킬은/플러그인은/컬렉션은 ~ 입니다." (무엇인지)
  - 2문장: "~ 때문에 ~에 적합합니다/잘 어울립니다." (누구에게, 왜)
- `tags`: 소문자 kebab-case, `#` 접두, 2~4개. README 핵심 키워드에서 뽑는다.
- `stars`·`forks`: `scripts/sync-stars.py`가 주기적으로 갱신하므로 추가 시점 값이면 충분하다.
- `editors_pick`: 게이트 5를 통과하고 승인받은 경우에만 `true`. 그 외는 필드 자체를 생략.
- `lang`: 사용자가 명시 요청한 경우에만. 기본은 넣지 않는다.
- 저자 소셜(페북·링크드인·X)은 카드가 아니라 최상위 `authors` 맵에 owner 단위로 저장한다. 같은 저자의 모든 카드에 자동 적용된다.

## 7. 파일 동기화 룰

카드 데이터는 두 곳에 이중으로 존재한다. 한쪽만 고치면 `file://` 프리뷰가 어긋난다.

- `docs/skills.json`: 정본.
- `docs/index.html`의 `<script type="application/json" id="skills-data">` 인라인 블록: `file://` 폴백용. 정본과 항상 동일하게 유지한다.

카드를 추가·제거·수정할 때 **두 곳을 함께** 바꾼다. `scripts/sync-inline.py`가 skills.json을 인라인 블록에 복사해 주며, `.githooks/pre-commit`이 커밋 직전에 호출한다.

## 8. 검증 (변경 후 필수)

변경 후 아래를 통과해야 마무리한다.

```bash
python3 - <<'PY'
import re, json, pathlib
j = json.load(open('docs/skills.json'))
html = pathlib.Path('docs/index.html').read_text()
inline = json.loads(re.search(r'id="skills-data">(.*?)</script>', html, re.DOTALL).group(1))
assert len(j['skills']) == len(inline['skills']), f"mismatch: json={len(j['skills'])} inline={len(inline['skills'])}"
assert len(j['skills']) == 150, f"total must be 150, got {len(j['skills'])}"
print("OK:", len(j['skills']))
PY
```

- 확인 항목: skills.json JSON 유효성, 인라인 블록 유효성, 두 배열 길이 일치, 총량 150.
- 150 복원 작업이 진행 중인 과도기에는 총량 assert가 실패할 수 있다. 그 경우 길이 일치까지만 확인하고, 총량 미달 상태임을 사용자에게 보고한다.

## 9. 이 문서의 갱신 기준

- 게이트·기준·룰이 실제로 바뀌었을 때만 고친다. 절차 개선은 커맨드 파일에서 한다.
- 규칙을 바꾸면 같은 커밋에서 `MEMORY.md`에 결정 기록(날짜, 무엇을, 왜)을 남긴다.
- 이 문서에 개별 카드 이름이나 개수 통계를 넣지 않는다. 그런 값은 `skills.json`이 정본이고 여기 적으면 금방 낡는다.
