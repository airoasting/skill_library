사용자가 제공한 인자: `$ARGUMENTS`

이 커맨드의 책임은 **이 라이브러리(`skills.json`)에는 아직 없는** 스킬·플러그인·MCP 후보를 GitHub에서 찾아, `add-skill.md`의 에디터픽 평가 기준(효용성·대표성·신뢰성)으로 점수를 매겨 추천하는 것이다. 이미 등록된 카드는 절대 다시 추천하지 않는다 — 그건 `/find-skill`의 영역.

## 0. 인자 파싱

- 따옴표·줄바꿈을 합쳐 `query`로 본다. 비어 있으면 어떤 일을 하고 싶은지 한 줄로 다시 묻고 종료.
- 옵션:
  - `--n <int>`: 추천 개수. 기본 3, 최대 6. 범위를 벗어나면 1~6로 클램프.
  - `--cat <category-id>`: 결과를 특정 카테고리로 한정. 카테고리 id는 런타임에 `skills.json`의 `categories[].id`에서 읽어 검증(하드코딩 금지).
  - `--min <float>`: 평균 점수 컷오프. 기본 3.5. 이 값 미만은 "추천 안 함"으로 표시되더라도 결과에는 포함되지만, 더 좋은 후보가 있으면 자동 탈락.

## 1. 라이브러리 스냅샷 로드 (중복 회피용)

`Read`로 `skills.json`을 읽어 다음 두 셋을 만든다.

- `existing_repos`: 모든 카드의 `repo`를 `lower()`로 정규화한 set. 외부 검색 결과에서 이미 있는 저장소는 즉시 제외한다.
- `cat_defs`: `categories[].id` → `desc` 매핑. 후보의 카테고리 추정에 사용한다.

추출이 너무 크면 짧은 `python3 -c "..."`로 필요한 필드만 슬라이스해도 된다. 단 평가 판단은 LLM이 직접 한다.

## 2. 외부 검색

질의에서 핵심 작업·도메인 키워드를 2~4개 추출한 뒤, 다음 도구로 폭넓게 후보를 모은다.

### 2.1 gh search (1차)

`gh search repos`를 다음 변주로 여러 번 호출. 각 호출은 별도 후보군을 만들고 합집합으로 합친다.

```bash
gh search repos "<keywords>" --topic claude-code --sort stars --limit 30
gh search repos "<keywords>" --topic claude-skill --sort stars --limit 30
gh search repos "<keywords>" --topic mcp --sort stars --limit 30
gh search repos "<keywords>" --topic claude --sort updated --limit 30
gh search repos "<keywords>" --sort stars --limit 30
```

`--topic` 다섯 변주 중 일부는 결과가 비어도 OK. `--json fullName,description,stargazerCount,updatedAt,url,owner,primaryLanguage,repositoryTopics`로 메타 일괄 수집.

### 2.2 WebSearch / WebFetch (2차)

`gh`로 충분한 후보가 안 모이면(특히 토픽 태깅이 부실한 신생 저장소) WebSearch로 보강한다.

- "Claude skill <keyword>"
- "Claude Code plugin <keyword>"
- "MCP server <keyword>"
- 한국어 질의면 "<keyword> claude skill" 등 영문 변주도 같이 시도

WebSearch 결과 중 GitHub URL을 추출해 `gh repo view <owner>/<repo> --json ...`로 메타를 보강. `gh` 실패 시 WebFetch로 페이지를 읽어 stars·description·updated를 추출.

### 2.3 후보 정리

- `existing_repos`에 있는 건 즉시 제외.
- 같은 `<owner>/<repo>` 중복 제거.
- 명백한 비-스킬(예: 일반 라이브러리, 학습 자료 모음, 퀴즈 등)은 사전 제거.
- README가 비어 있거나 description이 한 줄도 없는 저장소는 보류(평가 근거를 만들 수 없음).
- 살아남은 후보가 6개 미만이면 키워드 변주를 한 번 더 돌려 보강.

## 3. 비즈니스 리더 적합성 사전 필터

이 라이브러리는 **비개발자 비즈니스 리더**가 타깃이라는 정체성을 가진다. 다음에 강하게 해당하면 후보에서 제외한다.

- README의 핵심 키워드가 PR·CI/CD·테스트·배포·코드 리뷰·git hook 등 개발 워크플로우에 한정.
- 사용에 YAML/JSON 워크플로우 작성, MCP 서버 직접 구성, 조건분기·재시도 같은 엔지니어 멘탈모델이 필요.
- "풀스택", "파이프라인", "런타임", "멀티 LLM 오케스트레이션"이 중심 가치.
- 도메인 전문성을 전제하는데, 그 도메인의 타깃이 비즈니스 리더가 아닌 실무자(개발자·분석가).

별 수가 많다고 자동 통과시키지 마라. 통과 못 한 후보는 점수 평가 자체를 생략한다.

## 4. 에디터픽 3축 평가 (각 5점)

살아남은 후보에 다음 점수를 매긴다. **점수와 근거를 출력에 그대로 노출**한다 — 사용자가 판단의 신뢰성을 직접 검증할 수 있어야 한다.

### 축 1. 효용성 (Usefulness) — 비즈니스 리더가 얼마나 빨리·얼마나 큰 효용을 뽑는가

| 점수 | 기준 |
|---|---|
| 5 | 설치 즉시 당일 업무 투입, 반복 사용 명확 |
| 4 | 설정 1~2단계 후 곧바로 가치 |
| 3 | 세팅 여러 단계, 쓸만하지만 허들 있음 |
| 2 | 특정 시나리오에서만 유용 |
| 0~1 | 개발자 멘탈모델 요구, 일반 리더는 막힘 |

### 축 2. 대표성 (Representativeness) — 해당 카테고리에서의 위상

| 점수 | 기준 |
|---|---|
| 5 | 이 카테고리를 설명할 때 가장 먼저 드는 카드, 상위 대안 없음 |
| 4 | 카테고리 내 명확한 차별점·우위 |
| 3 | 유의미한 차별점 있으나 경쟁 카드와 비등 |
| 2 | 비슷한 카드가 여럿, 특별한 우위 없음 |
| 0~1 | 동급·상위 카드가 다수 |

대표성 평가 시 **라이브러리 안의 같은 카테고리 기존 카드와 비교**하라. `cat_defs`와 같은 카테고리에 묶인 등록 카드 목록을 근거로 우위 또는 보완 관계를 제시.

### 축 3. 신뢰성 (Credibility) — 저장소 자체의 신뢰도

| 점수 | 기준 |
|---|---|
| 5 | Anthropic 공식 또는 그에 준하는 조직/저자 + ★수천 이상 + 활발한 유지보수 |
| 4 | ★1k+ AND 검증된 저자, 최근 활동 활발 |
| 3 | ★1k+ 또는 검증된 저자 중 하나, 유지보수 정상 |
| 2 | ★500~1k, 저자 미검증 |
| 0~1 | ★<500, 유지보수 정체 또는 저자 미검증 |

`updatedAt`이 1년 이상 정체이면 신뢰성에서 -1. README 기준으로 "deprecated" 명시되어 있으면 무조건 신뢰성 1점 이하.

### 평균과 추천 강도

| 평균 | 추천 강도 |
|---|---|
| 4.5+ | ✦✦✦ 강력 추천 — 라이브러리 정식 추가 가치 충분 |
| 4.0~4.4 | ✦✦ 추천 — 본인 워크플로우에 맞으면 도입 권장 |
| 3.5~3.9 | ✦ 검토 — 도입 전 README·이슈 한번 더 확인 |
| 3.5 미만 | 추천 안 함 — 점수와 함께 "탈락" 표기 |

`--min` 컷오프 미만 후보는 더 좋은 차상위 후보가 있으면 자동 교체. 후보가 정말 부족하면 컷오프 미만이라도 점수와 함께 표시하되 명시적으로 "기준 미달"이라고 적는다.

## 5. 출력 포맷

먼저 한 줄로 사용자가 무엇을 찾으려는지 네가 이해한 바를 요약한다.

> 이해한 작업: <한 문장>
> 검색 키워드: `<kw1>`, `<kw2>`, `<kw3>`
> 후보 풀: <원시 N개> → 적합성 필터 후 <M개>

그다음 평균 점수 내림차순으로 N개, 각 후보를 정확히 이 형식으로:

```
### N위. <repo name (Title Case)>  <추천 강도 ✦ 표기>
> <한국어 1문장 요약, desc 그대로 인용 금지·재서술>

- **효용성 X/5** — <근거 1~2줄, README/기능에서 인용 가능>
- **대표성 X/5** — <근거. 라이브러리의 같은 카테고리 카드와 비교>
- **신뢰성 X/5** — ★<stars> · 최근 업데이트 <YYYY-MM-DD> · 저자 <검증 메모>
- **평균: X.X / 5**

- 카테고리 매칭: `<category-id>` (해당 카테고리 desc와 어떻게 닿는지 한 줄)
- 링크: https://github.com/<owner>/<repo>
```

마지막 줄에 한 번에:

> 이 중 라이브러리에 정식 추가하고 싶은 후보가 있으면 [Issue](https://github.com/airoasting/skill_library/issues/new)에 "discovery: <repo>" 제목으로 남겨주세요. 큐레이터가 `/add-skill` 게이트로 검증 후 반영합니다.

### 0건일 때

검색을 두 번 변주했는데도 후보가 안 나오면 다음을 안내한다.

1. 어떤 키워드로 어디까지 봤는지 짧게 보고.
2. 질의를 쪼개거나 영문 변주(예: "Claude memory", "MCP browser")를 시도해 보길 권장.
3. 정말 새로운 영역이라면 라이브러리에 카테고리를 만들어 시작할 가치가 있는지 한 줄 의견.

## 6. 절대 하지 말 것

- 라이브러리에 **이미 있는** 카드를 다시 추천하지 마라. `existing_repos` 검사 게이트를 건너뛰지 말 것.
- 점수를 만들기 위해 README를 읽지 않은 채로 추측하지 마라. 평가 근거가 부실하면 그 후보는 "근거 부족"으로 표시.
- 라이브러리에 카드를 직접 추가하지 마라. 그건 큐레이터가 `/add-skill`로 진행한다.
- "추가하면 좋겠다"는 권유 외에 사용자가 묻지 않은 다음 행동을 강요하지 마라.
- 신뢰성 점수에 stars만 보고 만점 주지 마라. 활발한 유지보수·저자 검증을 같이 본다.

## 7. 자가 검증 (출력 직전)

- 추천한 모든 카드의 `<owner>/<repo>`가 `existing_repos`에 없는가?
- 점수 표기·평균·추천 강도(✦)가 5단계 기준에 부합하는가?
- 매칭 카테고리 id가 실제 `categories[].id`에 있는가?

문제 있으면 그 카드만 빼고 차상위 후보로 다시 채워라.
