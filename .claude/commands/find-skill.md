사용자가 제공한 인자: `$ARGUMENTS`

이 커맨드의 **단 하나의 책임**은 이 저장소의 `skills.json`에 등록된 스킬 중에서 사용자의 상황에 가장 잘 맞는 것을 골라 추천하는 것이다. 외부 검색·새 스킬 추천·신규 카드 추가는 절대 하지 마. 그 일은 `/add-skill`의 영역이다.

## 0. 인자 파싱

- 따옴표·줄바꿈 모두 자연어 질의로 합쳐 `query`로 본다. 비어 있으면 사용자에게 한 줄로 어떤 일을 하고 싶은지 다시 물어보고 종료.
- 옵션:
  - `--n <int>`: 추천 개수. 기본 3, 최대 8. 범위를 벗어나면 1~8로 클램프.
  - `--cat <category-id>`: 결과를 특정 카테고리로 한정. 카테고리 id 목록은 **런타임에** `skills.json`의 `categories[].id`에서 읽어 검증한다(하드코딩 금지). 일치하는 id가 없으면 가장 가까운 후보 2~3개를 제시하고 사용자에게 다시 묻는다.
  - `--meta build|growth|execution`: 메타 카테고리(`meta_categories`)로 한정. id 검증도 동일하게 런타임 조회.
  - `--pick`: `editors_pick: true`인 카드만 후보에 포함.
- 옵션이 충돌하면(`--cat`과 `--meta`가 다른 그룹을 가리킴 등) 그대로 교집합으로 좁힌다. 결과가 0개면 4단계의 "결과 없음" 흐름으로.

## 1. 데이터 로드

다음을 한 번에 처리한다(파일을 두 번 읽지 마):

- `Read`로 `skills.json`을 읽는다. 파일이 너무 커서 단일 호출에 안 들어오면, 메타·카테고리·스킬을 추출하기 위해 짧은 `python3 -c "..."` 한 줄로 필요한 필드만 슬라이스해 받아도 된다. 단, **추천 판단은 LLM이 직접** 한다. 점수 산출을 파이썬에 떠넘기지 마라.
- 추출 대상:
  - `meta_categories[]`: id, name, desc
  - `categories[]`: id, meta, name, emoji, desc
  - `skills[]`: category, name, repo, author, stars, desc, tags, lang, editors_pick

## 2. 매칭

`query`에서 사용자가 진짜로 원하는 **작업 의도(job-to-be-done)** 와 **제약**을 분리해 읽는다. "이메일 자동 분류해줘"는 작업 의도, "한국어로", "구글 워크스페이스에서"는 제약이다.

각 후보 스킬에 대해 머릿속으로 다음 점수를 매긴다(외부 출력은 안 해도 된다).

- **의도 적합도(0~5)**: `desc` + `tags` + `name`이 작업 의도와 얼마나 직접 닿는가.
- **카테고리 적합도(0~3)**: 그 스킬이 속한 `categories[].desc`가 사용자의 일하는 맥락과 얼마나 맞는가. 카테고리 정의 자체가 강한 신호다.
- **제약 충족(0~2)**: 한국어/한국 서비스, 비개발자 친화, 특정 도메인(법무·재무 등) 같은 제약을 충족하는가. 명백한 제약 위반(예: "한국 환경"인데 한국 무관)이면 큰 폭 감점.
- **신뢰 보정(0~1)**: 동점이 있을 때만 `editors_pick`이거나 stars 상위인 카드를 살짝 우선.

총점 상위 N개를 고른다. 단 다음 룰을 지킨다.

- **카테고리 다양성**: 상위 후보가 한 카테고리에 3개 이상 몰리면, 같은 카테고리에서는 최대 2개까지만 남기고 나머지는 다른 카테고리의 차상위 후보로 교체한다. 사용자가 보지 못한 옵션을 노출하기 위함.
- **`--cat` / `--meta` / `--pick` 옵션이 켜진 경우**에는 다양성 룰을 풀고 옵션 조건만 지킨다. 사용자가 이미 범위를 좁혔다는 뜻이기 때문.
- **중복 카드 회피**: 같은 `repo`는 한 번만.

## 3. 출력 포맷

먼저 한 줄로 사용자가 무엇을 하려는지 네가 이해한 바를 요약한다(번역체 금지, 자연스러운 한국어 한 문장).

> 이해한 작업: <한 문장>

그다음 추천 카드를 N개, 각 카드는 정확히 이 형식으로:

```
### N위. <name>  ✦  <category emoji> <category 한글명>
- 무엇: <스킬 한 줄 설명, 직접 만들지 말고 desc에서 핵심만 압축>
- 왜 이걸 골랐나: <사용자 질의와 어떻게 닿는지 1~2문장. 카테고리 정의·태그·desc 중 어디서 신호를 받았는지 구체적으로>
- 한계/주의: <명확히 알면 1줄, 아니면 생략>
- 링크: https://github.com/<owner>/<repo>  ·  ★<stars>
- 라이브러리에서 보기: https://airoasting-skill.vercel.app/?cat=<category-id>
```

마지막에 한 줄로 다음 행동을 제안:

> 더 좁혀 볼래요? `--cat <id>`로 카테고리 고정, `--pick`으로 에디터픽만, `--n 5`로 더 많이 볼 수 있어요.

### 결과가 0개일 때

옵션이 너무 좁거나 라이브러리에 해당 분야가 없는 상황이다. 다음 순서로 안내한다.

1. 무엇이 결과를 0으로 만들었는지 한 줄로 짚는다(예: "`--cat legal`로 좁혔는데 질의가 한국어 글쓰기에 가까웠어요").
2. 옵션을 풀고 다시 매칭한 차선 후보 2개를 같은 포맷으로 제시한다. 그 두 개에는 "(범위를 풀어서 본 결과)" 표식을 단다.
3. 그래도 정말 라이브러리에 없는 분야로 보이면, **새 스킬을 직접 추천하지 마라.** 대신 사용자에게 이렇게 안내한다.

   > 이 라이브러리에는 아직 그 영역의 카드가 없어요. GitHub에서 후보를 찾아 에디터픽 기준으로 평가받고 싶다면 `/discovery-skill <같은 질의>`를 써보세요. 좋은 후보가 보이면 큐레이터에게 추가 요청을 남기실 수 있습니다.

### 어떤 경우에도 하지 말 것

- 라이브러리 **밖**의 스킬·도구·서비스를 추천하지 마라. 이 커맨드의 신뢰는 "여기 있는 것만 본다"는 데서 온다.
- 점수·내부 룰·매칭 로직을 출력에 노출하지 마라. 사용자는 결과만 본다.
- 카드의 `desc`나 `tags`를 그대로 길게 인용하지 마라. 사용자의 질의에 맞춰 축약·재서술해라.
- 결과를 표로 만들지 마라(모바일 가독성 떨어짐). 위 마크다운 카드 형식을 지켜라.

## 4. 검증 (출력 직전)

다음 두 가지만 빠르게 자가 확인:

- 추천한 모든 카드가 실제로 `skills.json`에 존재하는가? 이름이 비슷해서 만든 가짜 카드는 즉시 제거.
- 각 카드의 `category-id`가 `categories[].id`에 실제로 있는가? URL이 깨지면 안 된다.

문제 있으면 그 카드만 빼고 다시 매칭해 채워라.
